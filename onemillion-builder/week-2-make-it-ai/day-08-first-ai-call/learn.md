## Part 1: How LLMs Actually Work (~10 min read)

Three minutes of theory you actually need.

**LLMs (Large Language Models) like Claude predict the next token.**

A "token" is roughly a word, sometimes a part of a word. Claude reads your prompt, then generates one token at a time, each one chosen as the most likely "next thing" given everything it's seen so far.

That's it. There's no thinking, planning, or reasoning happening in the human sense. It's pattern completion at a massive scale, on a model trained on enormous amounts of text.

**What this means for you as a builder:**

1. **The AI only knows what you tell it.** No memory between calls (unless you build memory). Every prompt is fresh.
2. **Quality of output depends on quality of prompt.** Vague prompt → vague output. The 80/20 of AI engineering is writing good prompts.
3. **Longer prompts cost more.** Every token costs money (a tiny amount, but it adds up). Don't pad your prompts.
4. **AI will hallucinate confidently.** Sometimes it makes things up. Your job: design prompts that minimize this (and verify outputs that matter).

## Part 2: The Five Elements Of A Good Prompt (~15 min read)

Every effective prompt has five elements. Skip one and your output drifts.

### 1. Role
**Who is the AI being?** Without this, the AI defaults to "helpful generic assistant." Useless.

Bad: (no role specified)
Good: *"You are a client communication assistant for freelance UX designers."*

### 2. Goal
**What is the AI trying to achieve?**

Bad: *"Write something."*
Good: *"Help the user draft a status update email to one of their freelance clients."*

### 3. Tone
**How should the output sound?**

Bad: (no tone specified — output sounds generic)
Good: *"Friendly but professional. Use contractions. Avoid corporate jargon. Sound like a thoughtful professional, not a customer service rep."*

### 4. Constraints
**What MUST be true about the output? What MUST NOT be true?**

Bad: (no constraints — output goes off the rails)
Good:
- *"Always reference the deliverable by name."*
- *"Always propose a specific next step + deadline."*
- *"Never make up details about the client or project. If unsure, ask the user."*
- *"Output must be 80-200 words."*

### 5. Format
**What does the output look like structurally?**

Bad: (no format — output is unpredictable)
Good: *"Output a 3-paragraph email. Paragraph 1: acknowledgement. Paragraph 2: status + what's outstanding. Paragraph 3: next step + deadline + signoff."*

## Part 3: Putting It Together — A Real Prompt (~5 min read)

Bad prompt:
> "Help me write an email to my client."

Good prompt:
> ```
> You are a client communication assistant for freelance UX designers.
>
> Your goal: help the user draft a professional, friendly status update
> email to one of their clients about a specific deliverable.
>
> Tone: friendly but professional. Use contractions. Avoid corporate
> jargon. Sound like a thoughtful person, not a robot.
>
> Constraints:
> - Always reference the deliverable by name
> - Always propose a specific next step with a date
> - Never invent details about the client or project — if you need info
>   the user didn't provide, ask
> - Output must be 80-200 words
>
> Format: 3 paragraphs.
> - P1: Brief acknowledgement (1 sentence)
> - P2: Status update + what's outstanding (2-4 sentences)
> - P3: Proposed next step with a specific date + warm sign-off
>
> The user will provide: deliverable name, client name, current status,
> and the date of their last update to this client.
> ```

Same AI. Same model. Different prompt. **The good prompt produces useful output 80% of the time. The bad prompt produces useful output ~10% of the time.**

This is why your AI Feature Spec from Day 7 matters. Your spec IS the source material for your prompt. If your spec is vague, your prompt will be vague.

## Part 4: Where The Prompt Lives In Your Code (~5 min read)

**The prompt does NOT live in the frontend.** Two reasons:

1. **Your API key would leak.** If the prompt + API call happens in the user's browser, anyone can open DevTools and steal your Anthropic key. They can then use it to spend YOUR money.

2. **You'd lose control of the prompt.** Users can tamper with frontend code. They might trick the AI into doing things you don't want.

**The prompt lives in your backend — a Next.js API route.**

The flow:
```
User clicks "Draft Update" button
  → Frontend calls /api/ai/draft-update with the user's context
    → API route reads user's context
    → API route constructs prompt (role + goal + tone + constraints + format)
    → API route calls Anthropic API with the prompt
    → AI generates response
    → API route returns the response to frontend
      → Frontend displays the response
```

The frontend never sees the prompt or the API key. Safe + secure.

## Common Mistakes (Today)

1. **Putting the API key in client code.** This is the #1 mistake. If you see `process.env.NEXT_PUBLIC_ANTHROPIC_API_KEY` in any client component, you've leaked your key. Fix: never use the `NEXT_PUBLIC_` prefix for AI keys. They go in API routes only.

2. **Vague prompt.** Goes back to Day 7. If your spec is vague, your prompt will be too.

3. **Skipping the system prompt.** Some builders pass the user's input directly to Claude without a system prompt. The output drifts. Always set the role + goal upfront.

4. **No error handling.** AI APIs fail sometimes (rate limits, timeouts, etc.). Your code should handle errors gracefully and show a useful message to the user.

5. **Forgetting to add the env var to Vercel.** Works locally. Breaks on production. Same trap as Day 5. Use the pre-deploy checklist from Day 5.
